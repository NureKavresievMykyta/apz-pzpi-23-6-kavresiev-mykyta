2 ЗАВДАННЯ
      Практичне заняття спрямоване на вивчення одного з шаблонів проєктування програмного забезпечення з класичного каталогу «Банди чотирьох» (GoF). У процесі виконання роботи необхідно обрати один із патернів, крім Singleton, дослідити його призначення, структуру, учасників, переваги, обмеження та приклади практичного застосування в програмних системах.
      За результатами виконаного завдання потрібно підготувати доповідь і презентацію українською мовою у науково-технічному стилі. Слайди мають бути читабельними, мати стримане оформлення та виразні номери в правому нижньому куті кожного слайда. Також необхідно підготувати приклад програмної реалізації обраного патерна, описати його роботу, додати хронологічний опис відеозапису доповіді та оформити звіт відповідно до встановлених вимог.
 	Для виконання практичного заняття було обрано поведінковий патерн Observer, який українською мовою перекладається як «Спостерігач». Цей шаблон застосовується в ситуаціях, коли зміна стану одного об’єкта повинна автоматично викликати оновлення кількох інших об’єктів без створення жорсткої залежності між ними.


3 ОПИС ВИКОНАНОЇ РОБОТИ
      3.1 Загальні відомості про шаблони проєктування
     Шаблони проєктування програмного забезпечення являють собою узагальнені рішення типових задач, які часто виникають під час розроблення об’єктно-орієнтованих програмних систем. Вони не є готовими бібліотеками або фрагментами коду, а описують перевірені підходи до побудови архітектури, розподілу відповідальності між класами та організації взаємодії між об’єктами.
     Використання патернів допомагає зробити програмну систему більш зрозумілою, підтримуваною та придатною до розширення. Крім того, патерни формують спільну професійну мову для розробників, оскільки назва шаблона одразу вказує на певний спосіб розв’язання архітектурної задачі.
     Класичний каталог GoF містить 23 шаблони проєктування, які поділяються на три основні групи. Породжувальні патерни відповідають за створення об’єктів і дають змогу відокремити процес створення від конкретної реалізації. До них належать Factory Method, Abstract Factory, Builder, Prototype та Singleton. Структурні патерни описують способи поєднання класів і об’єктів у більші структури. До цієї групи входять Adapter, Bridge, Composite, Decorator, Facade, Flyweight та Proxy. Поведінкові патерни описують механізми взаємодії між об’єктами, передавання запитів і розподіл обов’язків. До них належать Chain of Responsibility, Command, Iterator, Mediator, Memento, Observer, State, Strategy, Template Method та Visitor.
     Предметом цієї роботи є патерн Observer. Він належить до групи поведінкових патернів, оскільки визначає спосіб взаємодії між об’єктами під час зміни стану одного з них.
      3.2 Призначення патерна Observer
     Патерн Observer визначає залежність типу «один до багатьох» між об’єктами. Його основна ідея полягає в тому, що один об’єкт виконує роль джерела змін, а інші об’єкти підписуються на ці зміни та автоматично отримують повідомлення після оновлення стану.
     Об’єкт, за яким спостерігають, називається суб’єктом. Він зберігає список підписників і повідомляє їх про зміну свого стану. Об’єкти, які реагують на ці зміни, називаються спостерігачами. Вони реалізують спільний інтерфейс, через який суб’єкт може звернутися до кожного з них, не знаючи їхніх конкретних класів.
     Необхідність використання патерна Observer виникає тоді, коли одна подія повинна викликати декілька незалежних реакцій. Наприклад, у медичній інформаційній системі додавання нового результату обстеження може одночасно оновити інтерфейс лікаря, надіслати повідомлення пацієнту та створити запис у журналі аудиту. Якщо всі ці дії викликати напряму з одного класу, система швидко стане жорстко зв’язаною, а будь-яке розширення вимагатиме зміни вже наявного коду.
     Патерн Observer усуває цю проблему. Суб’єкт лише повідомляє про зміну стану, а кожен спостерігач самостійно визначає власну реакцію. Завдяки цьому нові компоненти можна додавати без зміни основної логіки джерела події.
     Патерн доцільно застосовувати в таких випадках: коли зміна стану одного об’єкта повинна впливати на декілька інших об’єктів; коли кількість підписників може змінюватися під час виконання програми; коли потрібно зменшити залежність між компонентами; коли система працює з подіями, повідомленнями або оновленням інтерфейсу користувача.
      3.3 Структура патерна Observer
     Структура патерна Observer складається з чотирьох основних учасників. Subject - це інтерфейс або базовий клас об’єкта, за яким спостерігають. Він визначає методи для додавання, видалення та повідомлення спостерігачів. ConcreteSubject - конкретний суб’єкт, який зберігає певний стан і повідомляє підписників після його зміни. Observer - інтерфейс спостерігача, який визначає метод оновлення. ConcreteObserver - конкретна реалізація спостерігача, яка виконує власну дію після отримання повідомлення.
     Взаємодія між учасниками відбувається за простою послідовністю. Спочатку конкретний спостерігач реєструється у суб’єкта за допомогою методу підписки. Після цього суб’єкт зберігає його у внутрішньому списку. Коли стан суб’єкта змінюється, він викликає метод повідомлення, який послідовно звертається до всіх зареєстрованих спостерігачів. Кожен спостерігач отримує повідомлення та виконує власну реакцію.
     Важливою особливістю патерна є те, що суб’єкт не залежить від конкретних класів спостерігачів. Він працює тільки з абстрактним інтерфейсом Observer. Саме це дозволяє додавати нові реакції на подію без зміни класу суб’єкта. Така організація відповідає принципу відкритості до розширення та закритості до модифікації.
     У спрощеному вигляді структура патерна має такий зміст: суб’єкт зберігає список спостерігачів, надає методи attach і detach для керування підпискою та метод notify для повідомлення. Спостерігач має метод update, який викликається після зміни стану суб’єкта.
      3.4 Реалізація патерна Observer мовою Java
     Для демонстрації роботи патерна Observer було реалізовано приклад мовою Java. Як предметну область обрано медичну інформаційну систему, у якій після додавання нового результату обстеження повинні відреагувати кілька незалежних компонентів.
     У прикладі створено інтерфейс Observer з методом update. Цей метод отримує текст нового результату обстеження та виконує відповідну дію. Клас MedicalResultSubject виконує роль суб’єкта. Він містить список спостерігачів, методи attach і detach для додавання та видалення підписників, а також метод notifyObservers, який повідомляє всі зареєстровані об’єкти про зміну результату.

     Конкретними спостерігачами є класи DoctorPanel, PatientNotificationService та AuditLog. Клас DoctorPanel імітує оновлення панелі лікаря після появи нового результату. Клас PatientNotificationService відповідає за повідомлення пацієнта. Клас AuditLog створює запис у журналі аудиту. Кожен із цих класів реалізує інтерфейс Observer, але має власну логіку реакції.
     У методі main створюється об’єкт MedicalResultSubject, після чого до нього додаються три спостерігачі. Коли викликається метод setResult, стан суб’єкта змінюється, а всі підписники автоматично отримують повідомлення. Такий приклад демонструє головну перевагу патерна: клас MedicalResultSubject не містить прямої залежності від конкретних модулів, які реагують на подію.

      3.5 Переваги та недоліки патерна Observer
     Основною перевагою патерна Observer є зменшення зв’язаності між компонентами програмної системи. Суб’єкт не знає конкретних класів спостерігачів, а лише працює зі спільним інтерфейсом. Завдяки цьому окремі модулі можна додавати, змінювати або видаляти з мінімальним впливом на інші частини програми.
     Ще однією перевагою є гнучкість розширення. Якщо в системі потрібно додати нову реакцію на подію, достатньо створити новий клас-спостерігач і підписати його на суб’єкт. При цьому не потрібно змінювати код класу, який генерує подію. Це спрощує супровід системи та зменшує ризик появи помилок під час розширення функціональності.
     Патерн також добре підходить для подієво-орієнтованих систем, графічних інтерфейсів, мобільних застосунків, систем сповіщень, журналювання та оброблення змін даних. У таких системах часто виникає потреба в автоматичному оновленні кількох компонентів після однієї події.
     Водночас патерн має певні недоліки. Якщо кількість спостерігачів стає великою, може бути складно простежити, які саме об’єкти реагують на зміну стану. Це ускладнює відлагодження програми. Крім того, порядок повідомлення спостерігачів не завжди є очевидним, якщо він спеціально не визначений у реалізації.
     Окрему увагу потрібно приділяти відписці спостерігачів. Якщо об’єкт більше не використовується, але залишається у списку підписників, це може спричинити зайві виклики або проблеми з використанням пам’яті. Тому під час реалізації патерна потрібно чітко контролювати життєвий цикл підписників.
     Отже, Observer є ефективним інструментом для побудови слабозв’язаних систем, але його потрібно застосовувати там, де справді існує потреба в механізмі підписки та повідомлення.


4 ВИСНОВКИ
     У результаті виконання практичного заняття було досліджено поведінковий патерн проєктування Observer із класичного каталогу GoF. Проаналізовано його призначення, структуру, учасників та особливості взаємодії між об’єктами.
     Під час роботи встановлено, що патерн Observer доцільно застосовувати у випадках, коли зміна стану одного об’єкта повинна автоматично викликати реакцію кількох інших об’єктів. Його використання дозволяє уникнути прямої залежності між джерелом події та компонентами, які на неї реагують.
     Практичне застосування патерна продемонстровано на прикладі медичної інформаційної системи мовою Java. У реалізації клас MedicalResultSubject виконує роль суб’єкта, а DoctorPanel, PatientNotificationService та AuditLog є конкретними спостерігачами. При зміні результату обстеження всі підписники отримують повідомлення та виконують власні дії.
     Визначено основні переваги патерна: слабка зв’язаність між компонентами, зручність розширення, можливість динамічного додавання підписників і відповідність подієвій моделі взаємодії. Також проаналізовано недоліки: складність відлагодження у великих системах, неочевидний порядок повідомлення спостерігачів і необхідність контролю відписки.
     У межах практичного заняття було підготовлено презентацію, текст доповіді, приклад програмного коду та звіт. Отримані результати підтверджують, що використання патернів проєктування допомагає створювати більш гнучкі, масштабовані та підтримувані програмні системи.

package ua.nure.kavresiev.pz;

import java.util.ArrayList;
import java.util.List;

interface Observer {
    void update(String result);
}

class MedicalResultSubject {
    private final List<Observer> observers = new ArrayList<>();
    private String result;

    public void attach(Observer observer) {
        observers.add(observer);
    }

    public void detach(Observer observer) {
        observers.remove(observer);
    }

    public void setResult(String result) {
        this.result = result;
        notifyObservers();
    }

    private void notifyObservers() {
        for (Observer observer : observers) {
            observer.update(result);
        }
    }
}

class DoctorPanel implements Observer {
    @Override
    public void update(String result) {
        System.out.println("Панель лікаря оновлено. Новий результат: " + result);
    }
}

class PatientNotificationService implements Observer {
    @Override
    public void update(String result) {
        System.out.println("Пацієнту надіслано повідомлення про новий результат.");
    }
}

class AuditLog implements Observer {
    @Override
    public void update(String result) {
        System.out.println("Запис у журналі аудиту: додано результат - " + result);
    }
}

public class ObserverDemo {
    public static void main(String[] args) {
        MedicalResultSubject medicalResult = new MedicalResultSubject();

        Observer doctorPanel = new DoctorPanel();
        Observer notificationService = new PatientNotificationService();
        Observer auditLog = new AuditLog();

        medicalResult.attach(doctorPanel);
        medicalResult.attach(notificationService);
        medicalResult.attach(auditLog);

        medicalResult.setResult("Гострота зору: 0.8");
    }
}
