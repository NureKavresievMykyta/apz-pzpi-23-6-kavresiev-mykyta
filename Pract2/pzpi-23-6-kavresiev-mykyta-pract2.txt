2 ЗАВДАННЯ
     Метою практичного завдання є дослідження архітектури відомої програмної системи та формування навичок аналізу архітектурних рішень, які застосовуються у сучасних розподілених програмних платформах.
     У межах практичного завдання необхідно обрати відому програмну систему, яка не повторює теми інших студентів, проаналізувати її призначення, основні функції, архітектурний стиль, логічну структуру, ключові компоненти, підходи до взаємодії сервісів, обробки даних, масштабування та забезпечення надійності.
     Для виконання роботи було обрано програмну систему Spotify. Вибір цієї системи обумовлений тим, що Spotify є великою потоковою мультимедійною платформою, яка одночасно обслуговує значну кількість користувачів, працює з великими обсягами аудіоконтенту, використовує персоналізацію, рекомендаційні алгоритми, мікросервісний підхід та подієву обробку даних.


3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про програмну систему Spotify
     Spotify - це програмна система потокового доступу до аудіоконтенту, яка забезпечує пошук, відтворення, каталогізацію та персоналізоване надання музики, подкастів і пов’язаних мультимедійних матеріалів. Система використовується слухачами, авторами контенту, рекламодавцями, аналітичними підрозділами та внутрішніми інженерними командами.
     Основними функціями Spotify є відтворення аудіоконтенту, пошук треків, альбомів, виконавців і подкастів, створення та редагування плейлистів, формування персональних рекомендацій, підтримка офлайн-доступу для користувачів Premium, збирання аналітичних даних про прослуховування, а також забезпечення роботи рекламних і платіжних механізмів.
     Spotify належить до класу потокових мультимедійних платформ. Також систему можна розглядати як SaaS-сервіс із персоналізацією, розподілену хмарну систему та data-driven продукт, у якому дані про поведінку користувачів активно використовуються для покращення якості сервісу.
     Особливістю Spotify є те, що система повинна одночасно підтримувати велику кількість клієнтських застосунків, забезпечувати швидке відтворення контенту, обробляти події користувацької активності, формувати рекомендації та підтримувати стабільну роботу великої кількості внутрішніх сервісів.
     3.2 Обґрунтування вибору системи
     Spotify було обрано для аналізу через складність її архітектури та наявність значної кількості архітектурних рішень, характерних для сучасних програмних систем великого масштабу.
     По-перше, Spotify працює з великим потоком користувацьких запитів. Користувачі можуть одночасно запускати треки, ставити їх на паузу, пропускати композиції, додавати контент до плейлистів, виконувати пошук, змінювати пристрої відтворення та взаємодіяти з рекомендаціями. Усі ці дії повинні оброблятися швидко й узгоджено.
     По-друге, система підтримує різні типи клієнтів: мобільні застосунки, вебплеєр і десктопні застосунки. Вони мають різні технічні обмеження, однак повинні працювати з єдиною серверною платформою та забезпечувати синхронізацію стану користувача.
     По-третє, Spotify активно використовує подієву обробку та аналітичні механізми. Дані про прослуховування, пошук, вподобання та пропуски треків використовуються для бізнес-аналітики, A/B-тестування, побудови рекомендацій та оцінювання якості продукту.
     По-четверте, Spotify є прикладом системи, у якій важливу роль відіграє платформна інженерія. Велика кількість сервісів потребує інструментів для каталогізації, документування, стандартизації та підтримки внутрішньої розробки. Одним із таких інструментів є Backstage.
     3.3 Архітектурний стиль системи
     Архітектура Spotify має розподілений характер і базується на мікросервісному підході. Система не є єдиним монолітним застосунком, а складається з набору окремих сервісів, кожен із яких відповідає за певну частину предметної області.
     Мікросервісний підхід передбачає поділ системи на незалежні компоненти з окремими зонами відповідальності. Наприклад, окремі сервіси можуть відповідати за користувачів, каталог контенту, відтворення, пошук, плейлисти, рекомендації, рекламу, оплату та аналітику.
     Такий архітектурний стиль є доцільним для Spotify з кількох причин. По-перше, він дозволяє масштабувати окремі частини системи незалежно одна від одної. Якщо зростає навантаження на пошук або відтворення, можна збільшити ресурси саме для відповідних сервісів, не масштабуючи всю систему повністю.
     По-друге, мікросервіси спрощують роботу інженерних команд. Окремі команди можуть розробляти, тестувати й розгортати свої сервіси незалежно, що прискорює розвиток продукту.
     По-третє, такий підхід сприяє ізоляції збоїв. Помилка в одному сервісі не повинна повністю зупинити роботу всієї платформи. Наприклад, тимчасова недоступність рекомендаційного сервісу не повинна заважати користувачеві відтворювати музику або відкривати власні плейлисти.
     Крім мікросервісної архітектури, у Spotify застосовується подієво-орієнтований підхід. Події користувачів і системні події передаються до потоків обробки та використовуються для аналітики, моніторингу, рекомендацій і бізнес-метрик.
     3.4 Загальна логічна архітектура Spotify
     Логічну архітектуру Spotify можна умовно поділити на чотири основні рівні: клієнтський рівень, рівень API та шлюзів, сервісний рівень, а також рівень даних і подій.
     Клієнтський рівень включає мобільні застосунки, вебплеєр і десктопні клієнти. Саме через цей рівень користувач взаємодіє із системою. Клієнти відповідають за інтерфейс, запуск відтворення, кешування, роботу з обліковим записом, відображення плейлистів, пошук і передачу подій користувацької активності.
     Рівень API та шлюзів виконує функцію проміжного шару між клієнтськими застосунками та внутрішніми сервісами. На цьому рівні реалізуються автентифікація, авторизація, маршрутизація запитів, обмеження частоти запитів та первинна перевірка звернень до системи. API Gateway дозволяє приховати внутрішню складність платформи від клієнтів і надати їм уніфіковану точку доступу.
     Сервісний рівень складається з доменних сервісів. До нього можуть належати Playback Service, Catalog Service, Playlist Service, Search Service, Recommendation Service, User Service, Ads Service та Billing Service. Кожен із цих сервісів виконує окрему бізнес-функцію.
     Рівень даних і подій забезпечує зберігання метаданих, користувацьких даних, інформації про контент, подій активності, аналітичних показників і ознак для машинного навчання. На цьому рівні формується основа для рекомендацій, звітності та подальшого розвитку продукту.

     3.5 Клієнтський рівень системи
     Клієнтський рівень Spotify представлений кількома типами застосунків. Основними з них є мобільний застосунок, вебплеєр і десктопний клієнт.
Мобільний застосунок забезпечує відтворення музики, роботу з акаунтом, кешування контенту, push-сповіщення, керування плейлистами та взаємодію з рекомендаціями. Для мобільного клієнта важливими є стабільність роботи в умовах нестабільного інтернет-з’єднання, ефективне використання пам’яті та підтримка фонової роботи.
     Вебплеєр забезпечує доступ до Spotify через браузер. Він дає змогу користувачеві виконувати пошук, слухати музику, керувати плейлистами й отримувати рекомендації без встановлення окремого програмного забезпечення. Вебплеєр взаємодіє з внутрішньою платформою через API.
     Десктопний клієнт забезпечує стабільне відтворення, локальне кешування, інтеграцію з операційною системою та синхронізацію стану користувача. Він може забезпечувати ширші можливості взаємодії з пристроями та локальними ресурсами порівняно з вебверсією.
     Клієнтські застосунки не повинні напряму працювати з внутрішніми базами даних. Такий підхід зменшує ризики безпеки, спрощує зміну внутрішньої архітектури та дозволяє централізовано контролювати доступ до даних. Усі клієнтські звернення проходять через API, а дії користувачів можуть передаватися до системи як події.
     3.6 Мікросервісний підхід у Spotify
     Мікросервісний підхід є одним із ключових архітектурних рішень Spotify. Його сутність полягає в тому, що складна система розбивається на велику кількість окремих сервісів, які взаємодіють між собою через API або подієві механізми.
     До типових доменних сервісів Spotify можна віднести User Service, Catalog Service, Playback Service, Playlist Service, Search Service, Recommendation Service, Ads Service та Billing Service.
     User Service відповідає за облікові записи користувачів, профілі, параметри доступу та персональні налаштування. Catalog Service забезпечує роботу з каталогом треків, альбомів, виконавців і подкастів. Playback Service відповідає за логіку запуску, контролю та синхронізації відтворення аудіоконтенту. Playlist Service забезпечує створення, редагування та зберігання плейлистів. Search Service відповідає за пошук контенту. Recommendation Service формує персоналізовані добірки та рекомендації. Ads Service і Billing Service пов’язані з рекламою, оплатою та підписками.
     Перевагою такого підходу є висока гнучкість розвитку системи. Окремі сервіси можна оновлювати без повної зупинки платформи. Крім того, кожен сервіс можна масштабувати залежно від реального навантаження. Це особливо важливо для системи, у якій різні функції мають різну інтенсивність використання.
     Недоліком мікросервісної архітектури є зростання складності взаємодії між компонентами. Потрібно контролювати мережеві виклики, залежності, версії API, моніторинг, логування та відмовостійкість. Саме тому в Spotify значну роль відіграють внутрішні платформні інструменти, зокрема Backstage.
     3.7 Дані та подієва обробка
     Spotify активно використовує дані про дії користувачів. Кожна взаємодія з платформою може бути представлена у вигляді події. До таких подій належать запуск треку, пауза, пропуск композиції, лайк, пошуковий запит, додавання до плейлиста та зміна пристрою відтворення.
     Подієва обробка виконує кілька функцій. По-перше, вона дозволяє фіксувати поведінку користувачів для подальшого аналізу. По-друге, вона забезпечує основу для рекомендаційних систем. По-третє, вона використовується для бізнес-аналітики, звітів, A/B-тестування та моніторингу якості продукту.
     Процес обробки подій можна описати кількома етапами. Спочатку подія створюється на клієнтському або серверному рівні. Далі вона структурується та проходить валідацію. Після цього подія передається до системи потокової обробки або черги повідомлень. На наступних етапах дані агрегуються, очищуються, збагачуються та використовуються для побудови аналітичних ознак.
     Наприклад, якщо користувач часто пропускає певний тип треків, ця інформація може бути використана для коригування рекомендацій. Якщо певний плейлист має високий рівень утримання користувачів, це може бути враховано під час формування нових добірок. Таким чином, подієва модель дозволяє Spotify не лише реагувати на поточні дії користувачів, а й формувати довгострокове розуміння їхньої поведінки.

     3.8 Персоналізація та рекомендаційна підсистема
     Персоналізація є однією з найважливіших функцій Spotify. Вона забезпечує формування індивідуального користувацького досвіду, коли кожен користувач отримує добірки, які відповідають його інтересам, історії прослуховувань і контексту використання.
     Вхідними даними для рекомендацій можуть бути історія прослуховувань, вподобання користувача, створені плейлисти, пошукові запити, пропуски треків, метадані композицій, альбомів і виконавців, а також контекст сесії. До контексту можуть належати пристрій, час використання, активність користувача та інші параметри.
     У спрощеному вигляді рекомендаційна підсистема складається з Recommendation Service, Feature Store або Data Platform, моделей машинного навчання та механізму ранжування результатів. Recommendation Service відповідає за надання рекомендацій у відповідь на запити клієнтів. Data Platform забезпечує накопичення та обробку даних. Моделі машинного навчання використовуються для оцінювання релевантності треків і добірок.
     Основна складність такої архітектури полягає у поєднанні двох вимог. З одного боку, система повинна швидко відповідати на запити користувача. Наприклад, персональні добірки мають завантажуватися майже одразу після відкриття застосунку. З іншого боку, моделі рекомендацій потребують регулярного навчання на великих масивах даних.
     Тому архітектура рекомендацій Spotify повинна бути одночасно продуктивною, масштабованою та здатною працювати з великими обсягами історичних і поточних даних.

     3.9 Відтворення контенту, масштабованість і надійність
     Відтворення аудіоконтенту є основною функцією Spotify, тому до цього процесу висуваються високі вимоги щодо швидкості, стабільності та доступності.
     Типовий процес відтворення можна описати так: користувач надсилає запит на запуск треку, система перевіряє права доступу, визначає необхідний медіафайл, виконує буферизацію та починає відтворення. Для користувача цей процес має виглядати як миттєва реакція на натискання кнопки, однак всередині системи відбувається взаємодія кількох компонентів.
     Масштабованість Spotify забезпечується горизонтальним збільшенням кількості екземплярів сервісів, балансуванням навантаження, кешуванням популярного контенту та розділенням операцій читання і запису. Особливо важливим є кешування популярних треків, оскільки воно дозволяє зменшити навантаження на основні сховища та прискорити доставку контенту користувачам.
     Надійність системи забезпечується через ізоляцію збоїв, повтор запитів, резервування компонентів, моніторинг, журналювання та систему сповіщень про помилки. Якщо один із компонентів тимчасово недоступний, система повинна або переключитися на резервний сценарій, або зберегти доступність основних функцій.
     Для кінцевого користувача головним показником якості є те, що музика починає відтворюватися швидко, не переривається під час прослуховування та коректно синхронізується між різними пристроями.
     3.10 Платформна інженерія та Backstage
     У великих мікросервісних системах важливою проблемою є керування значною кількістю сервісів, API, документації, залежностей і команд. Якщо таких компонентів стає дуже багато, інженерам складно швидко визначити власника сервісу, знайти актуальну документацію, зрозуміти залежності або створити новий компонент за єдиними правилами.
     Для вирішення цієї проблеми Spotify розробив Backstage. Backstage - це платформа для створення внутрішніх порталів розробника. Вона допомагає організувати каталог сервісів, технічну документацію, описи API, шаблони створення нових компонентів та інші інженерні інструменти.
     Backstage не є частиною інтерфейсу, який бачить звичайний користувач Spotify. Його призначення полягає у підтримці внутрішньої розробки та керуванні складністю програмної екосистеми.
     З архітектурної точки зору Backstage є важливим прикладом платформної інженерії. Він дозволяє зменшити хаос у великій кількості сервісів, підвищити прозорість системи, прискорити створення нових компонентів і забезпечити кращу стандартизацію інженерних процесів.

def recommend_tracks(user_history, catalog):
    preferred_genres = {}

    for track in user_history:
        genre = track["genre"]
        preferred_genres[genre] = preferred_genres.get(genre, 0) + 1

    top_genre = max(preferred_genres, key=preferred_genres.get)

    recommendations = [
        track for track in catalog
        if track["genre"] == top_genre
        and track["id"] not in [item["id"] for item in user_history]
    ]

    return recommendations[:10]

def select_playback_node(nodes):
    available_nodes = [
        node for node in nodes
        if node["status"] == "available"
    ]

    if not available_nodes:
        raise Exception("No available playback nodes")

    return min(
        available_nodes,
        key=lambda node: node["latency"] + node["load"] * 0.5
    )
