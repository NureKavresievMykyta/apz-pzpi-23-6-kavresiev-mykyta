1 ЗАВДАННЯ
     Мета роботи: розробити та описати front-end частину програмної системи підтримки обстеження пацієнтів офтальмологічного центру Oftalmika, зосередивши увагу на web-застосунку для лікаря та адміністратора, його взаємодії з backend-сервером, базою даних PostgreSQL та IoT-пристроєм Oftalmika Smart Vision Scanner.
     У межах лабораторної роботи необхідно:
1. Описати призначення та функціональність web-застосунку системи. 
2. Реалізувати окремий web-інтерфейс для лікаря та адміністратора. 
3. Описати прийняті інженерні рішення під час розробки web-частини. 
4. Описати взаємодію компонентів web-застосунку між собою та з іншими частинами системи. 
5. Розробити UML діаграму прецедентів для web-частини системи. 
6. Розробити UML діаграму компонентів. 
7. Побудувати додаткові UML-діаграми для кращого опису структури та поведінки web-застосунку, а саме діаграму пакетів та діаграму взаємодії. 
8. Описати реалізовані функції web-застосунку: вхід лікаря, реєстрацію лікаря через заявку, підтвердження лікаря адміністратором, календар прийомів, пошук пацієнта, проведення прийому, отримання IoT-результатів, перегляд медичних записів і рецептів. 
9. Перевірити роботу web-застосунку та сформулювати висновки.


2 ХІД ВИКОНАННЯ РОБОТИ
     2.1 Загальний опис web-застосунку
     Програмна система Oftalmika призначена для автоматизації основних процесів офтальмологічного центру. У межах цієї лабораторної роботи основний акцент було зроблено на розробці web-частини системи, яка забезпечує роботу лікаря та адміністратора.
     Web-застосунок виконує роль робочого кабінету для працівників офтальмологічного центру. Через нього лікар може переглядати календар прийомів, відкривати інформацію про пацієнтів, проводити обстеження, отримувати результати з IoT-пристрою, заповнювати медичну картку, завершувати прийом та переглядати рецепти. Адміністратор через окрему панель може переглядати заявки лікарів і підтверджувати їх доступ до системи.
     Архітектура web-частини побудована за клієнт-серверним принципом. Web-застосунок не працює напряму з базою даних. Усі операції виконуються через backend API, який обробляє запити, звертається до PostgreSQL та за потреби взаємодіє з IoT-сервісом.
     Загальна схема взаємодії має такий вигляд:
     React web-застосунок → REST API backend → PostgreSQL database
     Для сценарію отримання результатів обстеження схема розширюється:
     React web-застосунок → REST API backend → IoT-сервіс → REST API backend → PostgreSQL database
     Для реалізації web-частини було використано React. Запити до backend виконуються через Axios. Основні сторінки web-застосунку реалізовані як окремі компоненти, між якими користувач переходить за допомогою маршрутизації.
     2.2 Опис ролей користувачів web-застосунку
     У web-частині системи реалізовано дві основні ролі користувачів:
* лікар; 
* адміністратор. 
     Лікар працює з основною медичною частиною системи. Після входу до web-кабінету він бачить власний календар прийомів. Календар подано у вигляді тижневого розкладу, що дозволяє швидко оцінити завантаженість лікаря та відкрити потрібний прийом.
     Лікар може виконувати такі дії:
* переглядати календар прийомів; 
* перемикатися між тижнями; 
* фільтрувати записи за статусом; 
* оновлювати список записів без перезавантаження сторінки; 
* шукати пацієнта; 
* відкривати картку пацієнта; 
* проводити прийом; 
* отримувати дані з IoT-пристрою; 
* заповнювати медичну картку; 
* завершувати прийом; 
* переглядати медичні записи; 
* переглядати рецепти. 
     Адміністратор має окремий web-інтерфейс. Його головна функція полягає в контролі доступу лікарів до системи. Після реєстрації лікар не отримує доступ автоматично. Його акаунт створюється як неактивний, а адміністратор повинен підтвердити заявку.
     Такий підхід потрібен для того, щоб сторонні користувачі не могли самостійно отримати доступ до медичної інформації пацієнтів.
     2.3 Реалізація web-інтерфейсу лікаря
     Web-кабінет лікаря є основною частиною web-застосунку. Після успішної авторизації лікар потрапляє на головну сторінку кабінету, де відображається привітання, календар прийомів, пошук пацієнтів, блок медичних записів, блок рецептів та інформація про IoT-пристрій.
     Головним елементом кабінету лікаря є календар прийомів. Він дозволяє переглядати записи за тиждень. У календарі відображаються заплановані та завершені прийоми. Для зручності було реалізовано перемикання між тижнями, фільтрацію записів та кнопку оновлення записів.
     

Рисунок 1 – Кабінет лікаря з календарем прийомів

     Сторінка проведення прийому відкривається після вибору конкретного запису. На ній лікар бачить інформацію про пацієнта, блок результатів IoT-обстеження, форму медичної картки та кнопку завершення прийому.
     Під час прийому лікар може натиснути кнопку отримання результатів з IoT-пристрою. Якщо пристрій онлайн, система отримує дані та відображає їх у блоці результатів. Якщо пристрій офлайн, кнопка отримання результатів недоступна, а лікар бачить відповідне повідомлення.

Рисунок 2 – Сторінка проведення прийому лікарем

     Також у web-кабінеті реалізовано пошук пацієнта. Лікар може знайти пацієнта за іменем, прізвищем, телефоном або email. Після пошуку відкривається картка пацієнта, де відображаються основні дані, завершені прийоми, останні IoT-дані та актуальний рецепт.
     

Рисунок 3 – Пошук пацієнта у web-кабінеті лікаря
     2.4 Реалізація адміністративної частини
     Адміністративна частина web-застосунку призначена для керування доступом лікарів до системи. У межах поточної реалізації адміністратор може переглядати заявки лікарів і підтверджувати їх.
     Процес роботи адміністратора виглядає так:
1. Адміністратор входить до системи через сторінку авторизації. 
2. Система визначає роль користувача як ADMIN. 
3. Користувач переходить до адміністративної панелі. 
4. У панелі відображаються заявки лікарів. 
5. Адміністратор підтверджує заявку. 
6. Після підтвердження лікар може увійти до web-кабінету. 
     Такий механізм було реалізовано для того, щоб лікар не отримував доступ до медичних даних без підтвердження. Це важливо, оскільки система працює з персональною та медичною інформацією пацієнтів.
     

Рисунок 4 – Адміністративна панель із заявками лікарів


     2.5 Реалізація взаємодії web-застосунку з backend
     Для взаємодії web-застосунку з backend було створено окремий API-клієнт. У ньому вказано базову адресу backend-сервера:
import axios from "axios";

const api = axios.create({
    baseURL: "http://localhost:8080",
    headers: {
        "Content-Type": "application/json"
    }
});

export default api;

     Цей файл використовується всіма сторінками web-застосунку. Завдяки цьому адреса backend API зберігається в одному місці, а код сторінок залишається чистішим.
     У web-застосунку реалізовано запити до backend для таких операцій:
POST /api/web-auth/login
POST /api/web-auth/register-doctor
GET /api/doctor-appointments/calendar/{doctorId}
GET /api/doctor-appointments/visit/{visitId}
PUT /api/doctor-appointments/finish/{visitId}
GET /api/iot/status
POST /api/iot/scan
GET /api/doctor-patients/search
GET /api/doctor-medical-records
GET /api/doctor-prescriptions
GET /api/admin/doctors/pending
PUT /api/admin/doctors/{doctorUserId}/approve

     Під час входу web-застосунок надсилає логін і пароль до backend. Якщо дані правильні, backend повертає роль користувача. Після цього web-частина перенаправляє користувача або до кабінету лікаря, або до адміністративної панелі.
     Для збереження даних поточного користувача використовується localStorage. Там зберігаються ідентифікатор користувача, роль, логін та відображуване ім’я. Це дозволяє web-застосунку знати, хто саме працює із системою.
     2.6 Реалізовані функції web-застосунку
     У межах лабораторної роботи було реалізовано основну функціональність web-частини системи Oftalmika.
     Сторінка входу дозволяє лікарю або адміністратору авторизуватися в системі. Після входу система перевіряє роль користувача. Якщо це адміністратор, відкривається адміністративна панель. Якщо це лікар, відкривається кабінет лікаря.
     Реєстрація лікаря реалізована через окрему сторінку. Лікар вводить логін, пароль, ім’я, прізвище, спеціалізацію та номер кабінету. Після відправлення форми створюється заявка, яка очікує підтвердження адміністратором.
     Кабінет лікаря містить календар прийомів. На відміну від звичайного списку записів, календар дозволяє краще бачити розклад лікаря. Прийоми відображаються за днями та часом. Лікар може перемикатися між тижнями та фільтрувати записи.
Сторінка проведення прийому дозволяє лікарю працювати з конкретним пацієнтом.
На цій сторінці можна отримати IoT-результати, заповнити медичну картку та завершити прийом. Після завершення прийому пацієнт бачить його в історії обстежень у мобільному застосунку.
     Пошук пацієнта дозволяє лікарю швидко знайти потрібного пацієнта та переглянути його картку. У картці пацієнта відображаються персональні дані, завершені прийоми, останні IoT-результати та актуальний рецепт.
     Розділ рецептів призначений для перегляду рецептів, сформованих після завершених прийомів. Це дає лікарю можливість контролювати актуальні призначення пацієнта.


Рисунок 5 – Перегляд рецептів у web-застосунку

     2.7 Локалізація web-застосунку
     У межах лабораторної роботи було враховано вимогу щодо локалізації інтерфейсу. Основною мовою реалізованого web-застосунку є українська. Українською мовою підписані основні сторінки, кнопки, повідомлення, статуси та назви розділів.
     Формат дати та часу в системі подається у зрозумілому для користувача вигляді. Календар лікаря відображає записи за тижнем, а прийоми розподіляються за часовими слотами.
     Повне перемикання між українською та англійською мовами в поточній версії не реалізовано. Однак структура web-застосунку дозволяє додати інтернаціоналізацію в майбутньому через окремі словники перекладів.


3 ГРАФІЧНІ ІЛЮСТРАЦІЇ
     3.1 UML діаграма прецедентів
     UML діаграма прецедентів відображає основні ролі web-застосунку та функції, які вони виконують. У web-частині системи основними акторами є лікар та адміністратор.
     Лікар може увійти до web-кабінету, переглянути календар прийомів, знайти пацієнта, відкрити картку пацієнта, провести прийом, отримати IoT-результати, заповнити медичну картку, завершити прийом і переглянути рецепти.
     Адміністратор може увійти до адміністративної панелі, переглянути заявки лікарів і підтвердити доступ лікаря до системи.
     
     
Рисунок 6 – UML діаграма прецедентів web-застосунку Oftalmika
     3.2 UML діаграма компонентів
     UML діаграма компонентів описує структуру web-застосунку та його взаємодію з іншими частинами системи. Web-клієнт взаємодіє з backend через REST API. Backend, у свою чергу, звертається до PostgreSQL для роботи з даними та до IoT-сервісу для отримання результатів обстеження.
     Web-застосунок містить сторінки входу, реєстрації лікаря, кабінету лікаря, адміністративної панелі, проведення прийому, медичних записів і рецептів. Окремо виділено API-клієнт, який відповідає за HTTP-запити до backend.


Рисунок 7 – UML діаграма компонентів web-застосунку Oftalmika

     3.3 UML діаграма пакетів
     UML діаграма пакетів відображає логічну структуру web-частини системи. У web-застосунку можна виділити пакети сторінок, API-клієнта, стилів і маршрутизації.
     Пакет сторінок містить React-компоненти, які відповідають за основні екрани системи. API-пакет забезпечує взаємодію з backend. Пакет стилів визначає зовнішній вигляд web-інтерфейсу. Маршрутизація відповідає за перехід між сторінками.
     Такий поділ дозволяє зробити структуру web-застосунку зрозумілішою та спрощує його подальший розвиток.

Рисунок 8 – UML діаграма пакетів web-застосунку Oftalmika

     3.4 UML діаграма взаємодії
     UML діаграма взаємодії описує сценарій проведення прийому лікарем у web-застосунку. Процес починається з відкриття календаря прийомів. Далі лікар обирає потрібний запис і переходить на сторінку проведення прийому.
     Після відкриття прийому web-застосунок отримує інформацію про пацієнта та прийом з backend. Якщо IoT-пристрій доступний, лікар натискає кнопку отримання результатів. Backend звертається до IoT-сервісу, отримує дані обстеження та зберігає їх у базі даних. Після цього результати відображаються у web-інтерфейсі.
     Після завершення прийому backend змінює статус запису на COMPLETED, зберігає медичну картку та формує або оновлює рецепт.
     

Рисунок 9 – UML діаграма взаємодії під час проведення прийому


4 ПЕРЕВІРКА РОБОТИ СИСТЕМИ
     Перевірка роботи web-застосунку здійснювалася шляхом запуску backend-сервера, бази даних PostgreSQL та IoT-сервісу. Backend і база даних запускаються через Docker-контейнери, а web-застосунок запускається окремо в середовищі Node.js.


Рисунок 10 – Запущені Docker-контейнери системи

     Для перевірки web-частини було виконано вхід до системи під акаунтом лікаря. Після успішного входу відкривається кабінет лікаря, де відображається календар прийомів. У календарі можна переглядати записи, перемикатися між тижнями та відкривати прийом.
Рисунок 11 – Вхід до web-кабінету лікаря

     Також було перевірено сторінку проведення прийому. Лікар відкриває конкретний запис, бачить дані пацієнта, статус IoT-пристрою та кнопку отримання результатів. Якщо IoT-пристрій онлайн, результати автоматично надходять до web-інтерфейсу.
     
Рисунок 12 – Отримання IoT-результатів у web-кабінеті лікаря
     Після отримання результатів лікар може заповнити медичну картку та завершити прийом. У результаті статус прийому змінюється на COMPLETED, а рецепт стає доступним для перегляду.
     Для перевірки адміністративної частини було виконано вхід під акаунтом адміністратора. У панелі адміністратора відображаються заявки лікарів. Після підтвердження лікар отримує доступ до web-кабінету.

5 ВИСНОВОК
     У ході виконання лабораторної роботи було розроблено web-частину програмної системи Oftalmika. Основний акцент було зроблено на функціональності для лікаря та адміністратора.
     Було реалізовано web-кабінет лікаря, який містить календар прийомів, пошук пацієнтів, сторінку проведення прийому, блок отримання результатів з IoT-пристрою, медичні записи та рецепти. Також було реалізовано адміністративну панель, через яку адміністратор може підтверджувати заявки лікарів.
     Web-застосунок взаємодіє з backend-сервером через REST API. Дані зберігаються у PostgreSQL, а IoT-пристрій працює як окремий сервіс і передає результати офтальмологічного обстеження до системи. Це дозволяє показати не лише статичний web-інтерфейс, а повний сценарій роботи: від перегляду прийому лікарем до отримання IoT-результатів і завершення обстеження.
     Розроблені UML-діаграми дозволяють описати функціональність web-застосунку, його компоненти, логічну структуру та взаємодію з backend, базою даних і IoT-сервісом. Запропонована web-частина є придатною для подальшого розвитку системи, зокрема для розширення адміністративної панелі, додавання повної інтернаціоналізації, експорту даних, резервного копіювання та покращення управління користувачами.
