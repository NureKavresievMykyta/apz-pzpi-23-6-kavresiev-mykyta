1 ЗАВДАННЯ
     Мета роботи: дослідити та реалізувати спосіб масштабування backend-частини програмної системи підтримки обстеження пацієнтів офтальмологічного центру Oftalmika для роботи з більшим навантаженням.
     У межах лабораторної роботи необхідно:
1. Обрати стратегію масштабування backend-частини системи. 
2. Описати технічні рішення, які дають змогу масштабувати систему. 
3. Реалізувати запуск кількох копій backend-сервера. 
4. Налаштувати балансування запитів між backend-інстансами. 
5. Перевірити, що запити розподіляються між кількома копіями backend. 
6. Проаналізувати можливі вузькі місця системи при збільшенні навантаження. 
7. Описати перевірку роботи системи після масштабування. 


2 ХІД ВИКОНАННЯ РОБОТИ
     2.1 Загальний опис масштабування backend
     Програмна система Oftalmika складається з кількох основних частин: web-застосунку, мобільного застосунку, backend-сервера, бази даних PostgreSQL та IoT-сервісу Oftalmika Smart Vision Scanner.
     У звичайному режимі роботи всі клієнтські частини системи звертаються до одного backend-сервера. Backend приймає HTTP-запити, обробляє їх, звертається до бази даних PostgreSQL або IoT-сервісу та повертає результат користувачу.
     Початкова схема роботи системи мала такий вигляд:
	Web / Mobile / Postman – Backend – PostgreSQL
     Для лабораторної роботи було реалізовано масштабування backend-частини. Основна ідея полягає в тому, що замість одного backend-сервера запускається кілька однакових копій. Перед ними розміщується Nginx Load Balancer, який приймає запити від користувачів і розподіляє їх між доступними backend-контейнерами.
     Після масштабування схема роботи системи має такий вигляд:
	Web / Mobile / Postman - Nginx Load Balancer - backend-1, backend-2, backend-3 – PostgreSQL
     Така схема відповідає горизонтальному масштабуванню, оскільки кількість backend-серверів збільшується за рахунок запуску додаткових копій одного й того самого сервісу.
     2.2 Обрана стратегія масштабування
     Для системи Oftalmika було обрано горизонтальне масштабування backend-сервера.
     Горизонтальне масштабування означає, що система запускає кілька однакових копій backend-сервера. Кожна копія виконує однакові функції: обробляє запити авторизації, роботи з лікарями, пацієнтами, прийомами, медичними записами, рецептами та IoT-даними.
	У межах лабораторної роботи backend було запущено у трьох копіях: backend-1, backend-2, backend-3
     Усі три backend-контейнери працюють з однією спільною базою даних PostgreSQL. Це можливо, тому що backend не зберігає критичний стан у пам’яті конкретного контейнера. Основні дані системи зберігаються в базі даних.
     Користувач при цьому не звертається до конкретного backend-контейнера. Для нього існує один адрес: 
     http://localhost:8080
     На цьому адресі працює Nginx, який розподіляє запити між backend-контейнерами.
     2.3 Використання Nginx як Load Balancer
     Для балансування запитів між backend-контейнерами було використано Nginx. Він виконує роль проміжного сервера між клієнтом і backend-інстансами.
     Клієнтська частина системи надсилає запити на адресу локального сервера, цей запит спочатку потрапляє до Nginx. Далі Nginx передає його одному з доступних backend-контейнерів.
     Фрагмент конфігурації:
events {
    worker_connections 1024;
}

http {
    upstream oftalmika_backend {
        server backend:8080;
    }

    server {
        listen 80;

        location / {
            proxy_pass http://oftalmika_backend;
            proxy_http_version 1.1;

            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }
    }
}

     У цьому фрагменті upstream oftalmika_backend визначає групу backend-серверів, на які Nginx може передавати запити. Оскільки backend запускається через Docker Compose у кількох копіях, Nginx може спрямовувати запити на різні інстанси одного сервісу.
     2.4 Масштабування backend через Docker Compose
     Для запуску системи використовується Docker Compose. У ньому описано сервіси PostgreSQL, backend, IoT-пристрій та Nginx.
     Фрагмент конфігурації backend у docker-compose.yml:
backend:
  build:
    context: ./backend
    dockerfile: Dockerfile
  restart: always
  environment:
    DB_URL: jdbc:postgresql://postgres:5432/ophthalmica_db?options=-c%20TimeZone=UTC
    DB_USERNAME: ophthalmica_user
    DB_PASSWORD: ophthalmica_password
    IOT_DEVICE_URL: http://iot-device:8090
  expose:
    - "8080"
  depends_on:
    postgres:
      condition: service_healthy
    iot-device:
      condition: service_started
  networks:
    - ophthalmica_network

     У backend-сервісі не використовується container_name, оскільки це завадило б створенню кількох контейнерів одного сервісу. Docker Compose автоматично створює контейнери з назвами backend-1, backend-2, backend-3.
     
     Для запуску одного backend-сервера використовується команда:
     docker compose up --build --scale backend=1
     Для запуску трьох backend-серверів використовується команда:
     docker compose up --build --scale backend=3

     2.5 Endpoint для перевірки backend-інстанса
     Для наочної перевірки горизонтального масштабування було створено службовий endpoint:
     GET /api/system/instance
Він повертає інформацію про backend-контейнер, який обробив конкретний запит.
     Фрагмент коду контролера:
package ua.nure.kavresiev.optiview.controller;

import org.springframework.web.bind.annotation.CrossOrigin;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

import java.net.InetAddress;
import java.time.LocalDateTime;
import java.util.HashMap;
import java.util.Map;

@RestController
@CrossOrigin(origins = "*")
public class SystemController {

    @GetMapping("/api/system/instance")
    public Map<String, Object> getInstanceInfo() {
        Map<String, Object> response = new HashMap<>();

        String hostname = "unknown";

        try {
            hostname = InetAddress.getLocalHost().getHostName();
        } catch (Exception ignored) {
        }

        response.put("status", "OK");
        response.put("service", "Oftalmika Backend");
        response.put("currentBackendInstance", hostname);
        response.put("message", "This request was processed by one backend container behind Nginx Load Balancer");
        response.put("time", LocalDateTime.now().toString());

        return response;
    }
}

     Кожен Docker-контейнер має власний hostname. Тому під час багаторазового звернення до endpoint можна побачити, який саме backend-контейнер обробив кожен запит.

     2.7 Вертикальне масштабування backend
     Окрім горизонтального масштабування, для системи можливе також вертикальне масштабування. Воно полягає не в збільшенні кількості backend-контейнерів, а в збільшенні ресурсів одного backend-інстанса.
     Наприклад, одному backend-контейнеру можна виділити більше процесорного часу та оперативної пам’яті. Це дозволяє одному серверу обробляти більше запитів без запуску додаткових копій.
     Приклад налаштування ресурсів у Docker Compose:
deploy:
  resources:
    limits:
      cpus: "2.0"
      memory: 1024M
    reservations:
      cpus: "1.0"
      memory: 512M

     Таке рішення може бути корисним, якщо система працює на сервері з достатнім запасом ресурсів. Однак вертикальне масштабування має природні обмеження, тому що ресурси одного фізичного або віртуального сервера не можна збільшувати нескінченно.
     Для backend на Spring Boot також важливо налаштовувати пул підключень до бази даних. Наприклад:
       spring.datasource.hikari.maximum-pool-size=20
       spring.datasource.hikari.minimum-idle=5

     Ці параметри визначають, скільки одночасних підключень backend може використовувати для роботи з PostgreSQL. При збільшенні ресурсів backend-сервера можна також збільшити розмір пулу підключень, але це потрібно робити обережно, щоб не перевантажити базу даних.
     У межах лабораторної роботи основний акцент зроблено саме на горизонтальному масштабуванні, оскільки воно краще демонструється через кілька backend-контейнерів і load balancer.


3 АНАЛІЗ ВУЗЬКИХ МІСЦЬ
     Після масштабування backend-сервера навантаження на обробку HTTP-запитів розподіляється між кількома контейнерами. Це зменшує навантаження на один backend-інстанс і підвищує стійкість системи.
     Однак у системі залишаються ресурси, які можуть стати вузькими місцями при великому навантаженні.
     Першим вузьким місцем є база даних PostgreSQL. Усі backend-контейнери звертаються до однієї спільної бази даних. Якщо кількість запитів значно зросте, саме база даних може першою почати працювати повільніше, оскільки їй потрібно обробляти запити від усіх backend-інстансів.
     Другим вузьким місцем є пул підключень до бази даних. У Spring Boot для роботи з PostgreSQL використовується HikariCP. Якщо кількість одночасних запитів стане більшою, ніж кількість доступних підключень у пулі, частина запитів буде чекати на вільне підключення.
     Третім вузьким місцем можуть бути ресурси комп’ютера, на якому запускаються Docker-контейнери. Оскільки в лабораторній роботі система запускається локально, усі контейнери використовують ресурси одного комп’ютера: CPU, RAM і дискову підсистему.
     Четвертим вузьким місцем може бути IoT-сервіс. Якщо багато лікарів одночасно запускають отримання результатів з IoT-пристрою, окремий IoT-сервіс може стати точкою затримки.
     П’ятим можливим обмеженням є Nginx, але в межах лабораторної роботи він не є критичним вузьким місцем, оскільки для локальної демонстрації його продуктивності достатньо.


4 ПЕРЕВІРКА РОБОТИ СИСТЕМИ ПІСЛЯ МАСШТАБУВАННЯ
     Після запуску трьох backend-контейнерів було перевірено роботу основного API та web-частини системи.
     Спочатку було перевірено службовий endpoint:
     http://localhost:8080/api/system/instance
     Він показав, що запити обробляються різними backend-контейнерами. Після цього було відкрито web-кабінет лікаря. Система продовжила працювати так само, як і при одному backend-сервері. Лікар може увійти до системи, переглянути календар прийомів, відкрити сторінку проведення прийому, отримати IoT-результати та завершити прийом.


     Рисунок 3 – Web-кабінет лікаря після масштабування backend

     Також було перевірено, що IoT-сервіс залишається доступним. Backend-контейнери можуть звертатися до IoT-сервісу за внутрішньою Docker-адресою:
     http://iot-device:8090
     Таким чином, масштабування backend не порушило взаємодію з базою даних та IoT-компонентом.
5 ОПИС НАВАНТАЖУВАЛЬНОЇ ПЕРЕВІРКИ
     У межах лабораторної роботи не проводилося повноцінне навантажувальне тестування за допомогою JMeter або Gatling, оскільки основна мета роботи полягала в демонстрації можливості масштабування backend-сервера.
     Для базової перевірки було використано серію повторних API-запитів до endpoint:
     /api/system/instance
     Цей endpoint дозволив перевірити, що запити розподіляються між різними backend-інстансами. Порівняння режимів запуску наведено в таблиці 1.
     
     Таблиця 1 - Перевірка масштабування backend
№КонфігураціяКоманда запускуРезультат1Один backend-інстансdocker compose up --build --scale backend=1Запити обробляє один backend-контейнер2Три backend-інстансиdocker compose up --build --scale backend=3Запити розподіляються між трьома backend-контейнерами3Три backend + Nginxlocalhost:8080/api/system/instanceКористувач звертається до одного адреса, Nginx розподіляє запити
     У результаті перевірки було підтверджено, що збільшення кількості backend-інстансів не порушує роботу системи. Навпаки, така архітектура дозволяє обробляти запити різними копіями backend-сервера.
     Повноцінне навантажувальне тестування може бути виконане в подальшому з використанням JMeter, Gatling або Locust. У такому тестуванні можна виміряти кількість запитів на секунду, середній час відповіді та відсоток помилок для конфігурацій з одним, двома та трьома backend-інстансами.
6 ВИСНОВОК
     У ході виконання лабораторної роботи було реалізовано масштабування backend-частини програмної системи Oftalmika.
     Для масштабування було використано горизонтальний підхід. Backend-сервер було запущено у трьох однакових Docker-контейнерах: backend-1, backend-2 та backend-3. Перед ними було розміщено Nginx Load Balancer, який приймає запити на порту 8080 і розподіляє їх між доступними backend-інстансами.
     Для перевірки було створено службовий endpoint /api/system/instance, який повертає ідентифікатор backend-контейнера, що обробив запит. Під час багаторазового звернення до цього endpoint було отримано відповіді від різних backend-інстансів. Це підтверджує, що горизонтальне масштабування працює коректно.
     Також було розглянуто можливість вертикального масштабування backend-сервера шляхом збільшення ресурсів одного контейнера: CPU, RAM та кількості підключень до бази даних. Однак основний акцент у роботі зроблено саме на горизонтальному масштабуванні, оскільки воно краще підходить для web-систем і дозволяє збільшувати кількість backend-копій залежно від навантаження.
     У результаті було показано, що система може працювати не лише з одним backend-сервером, а й з кількома копіями backend, при цьому web-застосунок, база даних PostgreSQL та IoT-сервіс продовжують працювати коректно.
