1 ЗАВДАННЯ
     Мета роботи: підготувати коротку презентацію готового проєкту програмної системи Oftalmika та продемонструвати основні функції, які були реалізовані під час виконання комплексного проєкту.
     У межах лабораторної роботи необхідно:
1. Підготувати короткий опис проєкту. 
2. Описати заплановані та виконані функції. 
3. Підготувати матеріали презентації. 
4. Додати посилання на відео з демонстрацією роботи системи. 


2 ХІД ВИКОНАННЯ РОБОТИ
     2.1 Короткий опис проєкту
     Програмна система Oftalmika призначена для підтримки обстеження пацієнтів офтальмологічного центру. Система об’єднує мобільний застосунок пацієнта, web-кабінет лікаря, адміністративну панель, backend-сервер, базу даних PostgreSQL та IoT-сервіс.
     Основний сценарій роботи системи полягає в тому, що пацієнт записується до лікаря, лікар проводить прийом, отримує результати з IoT-пристрою, завершує обстеження, а пацієнт після цього може переглянути історію прийому та актуальний рецепт.
     2.2 Заплановані та виконані функції
     У межах проєкту було заплановано реалізувати основні функції для трьох ролей: пацієнта, лікаря та адміністратора.
     Було виконано:
* реєстрацію та вхід пацієнта; 
* запис пацієнта до лікаря; 
* перевірку зайнятих часових слотів; 
* web-кабінет лікаря; 
* календар прийомів; 
* пошук пацієнта; 
* проведення прийому; 
* отримання IoT-результатів; 
* формування рецепта; 
* перегляд історії обстежень; 
* адміністративне підтвердження лікарів; 
* запуск backend, бази даних та IoT-сервісу через Docker. 
     Не було повністю реалізовано промислову систему безпеки, сповіщення пацієнтів, експорт рецепта у PDF та інтеграцію з реальним медичним обладнанням. Ці функції можна додати під час подальшого розвитку системи.
     2.3 Матеріали презентації
     Для лабораторної роботи було підготовлено презентацію, у якій показано:
* короткий опис проєкту; 
* архітектуру системи; 
* виконані функції; 
* ключові можливості системи; 
* матеріали демонстрації; 
* використані технології; 
* подальші напрями розвитку. 


3 ФІНАЛЬНА ПРЕЗЕНТАЦІЯ
     «Тема мого проєкту - програмна система для офтальмологічних клінік. Проєкт розроблений для автоматизації роботи офтальмологічного центру, зокрема запису пацієнтів, проведення прийому, збереження результатів обстеження та перегляду рецепта пацієнтом».
     


Рисунок 1 – Титульний аркуш

     Метою роботи було створити програмну систему, яка об’єднує кілька важливих процесів: запис пацієнта, роботу лікаря, отримання IoT-результатів, ведення медичної картки та формування рецепта. Основний сценарій такий: пацієнт записується до лікаря, лікар проводить прийом, система отримує результати з IoT-пристрою, після чого пацієнт може переглянути рецепт.

Рисунок 2 -  Мета роботи

     Під час аналізу предметної області було визначено кілька проблем. Результати обстежень часто переносяться вручну, через що може виникати людська помилка. Також є проблема з накладанням записів, якщо час прийому не перевіряється автоматично. Окремо можна виділити те, що історія обстежень, рецепти та дані діагностичного обладнання можуть зберігатися розрізнено. Тому було вирішено створити єдину систему, яка об’єднує ці процеси.

Рисунок 3 – Аналіз проблеми


     У межах проєкту потрібно було реалізувати кілька основних частин системи. Для пацієнта створено мобільний застосунок. Для лікаря - web-кабінет із календарем прийомів. Для адміністратора - окрему панель підтвердження лікарів. Також було реалізовано backend API, базу даних PostgreSQL та IoT-сервіс Oftalmika Smart Vision Scanner, який імітує отримання результатів офтальмологічного обстеження.
     

Рисунок 4 – Постановка задачі та опис системи

     Для backend-частини було використано Java, Spring Boot і Spring Data JPA. Web-частина реалізована на React із використанням Axios для HTTP-запитів. Мобільний застосунок створено в Android Studio мовою Java, а для взаємодії з backend використано Retrofit. Дані зберігаються в PostgreSQL. IoT-сервіс реалізовано на Python з використанням FastAPI. Для запуску серверних компонентів використано Docker Compose.



Рисунок 5 – Вибір технологій розробки

     Архітектура системи побудована за клієнт-серверним принципом. Android-застосунок і React web-кабінет не звертаються до бази даних напряму. Усі запити проходять через backend API. Backend уже працює з PostgreSQL і, за потреби, звертається до IoT-сервісу. Такий підхід дозволяє централізовано контролювати бізнес-логіку та спрощує подальший розвиток системи.

Рисунок 6 – Архітектура створеного програмного забезпечення

     Під час розробки використовувались різні інструменти. Backend розроблявся в IntelliJ IDEA. Мобільний застосунок створювався в Android Studio. Для IoT-сервісу використовувався PyCharm. База даних переглядалася й наповнювалася через DBeaver. Docker Desktop використовувався для запуску контейнерів, а Postman - для перевірки backend API. Тобто кожен інструмент використовувався для окремої частини проєкту.

Рисунок 7 – Опис використаного програмного забезпечення

     Для всієї системи було обрано єдиний біло-сіро-синій стиль, тому інтерфейс виглядає стримано й підходить для медичної тематики. Мобільний інтерфейс зроблено простим для пацієнта, а web-кабінет лікаря має календарний вигляд прийомів. Також у системі є окрема сторінка проведення прийому, статус IoT-пристрою онлайн або офлайн, а незаповнені медичні поля відображаються як “не вказано”.


Рисунок 8 – Дизайн системи

     Як приклад реалізації можна розглянути передачу даних з IoT-пристрою. Під час прийому лікар натискає кнопку отримання результатів. Web-частина надсилає запит на backend, backend звертається до IoT-сервісу, отримує результати обстеження та зберігає їх у системі. Після цього лікар бачить показники, які можуть використовуватися для формування рецепта».



Рисунок 9 – Приклад реалізації
     У системі реалізовано кілька інтерфейсів. Пацієнт працює через мобільний застосунок: він може записатися до лікаря, переглянути історію обстежень і рецепт. Лікар працює через web-кабінет: він бачить календар, проводить прийом і отримує IoT-дані. Адміністратор підтверджує заявки лікарів, щоб доступ до системи отримували тільки перевірені користувачі».


Рисунок 10 – Інтерфейс користувача

     Під час тестування було перевірено основні сценарії роботи системи. Перевірялися реєстрація пацієнта, запис до лікаря, захист від зайнятого часу, вхід лікаря, отримання IoT-даних, завершення прийому та перегляд рецепта. Усі ці сценарії були виконані успішно, тому основний цикл роботи системи можна вважати перевіреним».
 
Рисунок 11 – Тестування

     У результаті було розроблено комплексну систему для офтальмологічних клінік яка складається з мобільного застосунку, web-кабінету, backend-сервера, бази даних та IoT-сервісу. Система демонструє повний цикл роботи: запис пацієнта, проведення прийому, отримання IoT-даних і перегляд рецепта. Надалі систему можна розширити: додати захищену авторизацію, сповіщення для пацієнтів, експорт документів та підключення реального медичного обладнання»
